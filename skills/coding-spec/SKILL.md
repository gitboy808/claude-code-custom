完成标准：在已理解问题的前提下，按七级阶梯选择最高可行档位，生成最短且正确的 diff。

## 七级阶梯

从第 1 档开始向下检查，停在第一档能用的方案：

1. **这个功能真的需要存在吗？** 投机性需求 = 跳过它，用一句话说明。（YAGNI）
2. **仓库里已经有了吗？** 现成的 helper、util、类型或模式 → 复用。先看再写；跨几屏重写已存在的东西是最常见的 slop。
3. **项目已装库能用更少代码解决吗？** 先看一眼构建清单——大多数项目都已经装着通用 helper。
4. **标准库 / 平台能覆盖吗？** `LocalDate.parse(...)`、`Files.lines(...)`、`Map.copyOf(...)`、`Stream.toList()`。默认兜底。在日期/线程/I/O 等安全敏感场景优先用平台保证（类型安全、线程安全）。
5. **项目框架原生抽象？** 框架惯用入口（参数绑定、校验、事务）优先于退到标准库。
6. **能一行搞定吗？** 一行。
7. **只有以上都不行时：** 写刚好够用的最小代码。

**平局裁决规则：** 当项目库与标准库同等简洁时，优先标准库（更少 import、更小 diff、无传递依赖风险）。当项目库显著更短优先它。

爬阶梯前必须先读完任务涉及的代码，端到端追真实流程。两档都行 → 选高的一档走人。在错的位置上做最小的改动不是最小化，是第二个 bug。

**Bug 修复 = 根因，而非症状。** 一个 bug 报告命名的是一个症状。在你动手编辑之前，grep 你即将触碰的函数的所有调用方。最小化的修复就是根因修复：在被共享的函数里加一个 guard，比每个调用方各加一个 guard 改得更小。只修补 ticket 命名的路径，会让每个兄弟调用方依然坏着。一次性修在所有调用都汇聚的地方。

## 规则

- 文件越少越好。最短且可用的 diff 胜出——但前提是你已经理解了问题。
- 复杂请求？交付最小化版本，并在同一次回复里质疑它："已做 X；Y 也能覆盖。需要完整的 X 吗？说一声。"不要在能默认的答案上犹豫不决。
- 两种标准库方案同等大小？选边界条件下正确的那种。最小化意味着写更少的代码，不是挑更脆弱的算法。
- 对于确实切掉了一个真实角落、带着已知天花板的主动简化（全局锁、O(n²) 扫描、naive 启发式），用 `yagni-coder:` 注释同时注明天花板与升级路径（`# yagni-coder: global lock, per-account locks if throughput matters`）。
- **功能关键位置必须留下日志线索。** 在关键执行路径、状态转换、外部依赖调用，以及异常或降级分支添加适量、可检索且带必要上下文的日志，供后续线上问题定位和还原现场。日志只包含诊断所需的业务标识、状态或结果。
- **功能关键位置必须添加 `//` 代码注释。** 注释解释为什么这样实现、受什么约束、规避了什么风险或做了什么取舍。只注释关键但不直观的决策。

## 最小化不可越过的边界

始终保留：信任边界的输入校验、防止数据丢失的错误处理、安全措施、可访问性基础、任何被明确要求的内容。用户坚持要完整版 → 直接构建，不要再争论。
